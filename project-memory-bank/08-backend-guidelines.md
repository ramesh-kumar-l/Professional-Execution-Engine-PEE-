# 08 — Backend Guidelines

Operational standards live in `claude/BACKEND.md` — not duplicated here. This file tracks the *product-specific* backend state.

## Current state

**Status: Phase 10 implemented, 2026-07-18 — the last phase currently defined.** TypeScript/NestJS modular monolith per `adr/0002`, npm workspaces.

## Module map

- **`/services/api`** (`@pee/api`) — composition root: `main.ts` (helmet, global `ValidationPipe`), `app.module.ts` (imports `ConfigModule`, `EventEmitterModule.forRoot()` (global, required for `@pee/planning`'s emitters and `@pee/execution`'s listeners), `ThrottlerModule` with a global `ThrottlerGuard`, `PrismaModule`, `AuthModule`, `OrganizationsModule` (Phase 10), `ProjectsModule`, `PlanningModule`, `ExecutionModule`, `SyncModule`, `AIModule`, `AnalyticsModule`), `HealthController` (`GET /health`).
- **`/services/auth`** (`@pee/auth`) — system of record for users/credentials, per `adr/0005`. `AuthController` (`/auth/register`, `/login`, `/refresh`, `/logout`, `/me`), `AuthService` (orchestration), `TokenService` (JWT access + opaque hashed refresh tokens), `PasswordService` (argon2), `AuditLogService`, `JwtStrategy`/`JwtAuthGuard`/`CurrentUser` decorator. Owns `User`, `RefreshToken`, `AuthAuditLog` — no other module reaches into this data. `JwtAuthGuard`, `CurrentUser`, and `CurrentUserPayload` are exported for reuse by other modules that need an authenticated caller (e.g. `@pee/projects`, `@pee/planning`, `@pee/organizations`). **Phase 10:** gained a `sso/` subtree (`SsoProvisioningService`, `SsoStatusController`, `oidc/provision.controller.ts`, `saml/` — `SamlService`/`SamlBridgeService`/`SamlController`/`redirect-allowlist.ts`) wired directly into `AuthModule`'s own providers/controllers rather than a nested `SsoModule`, and `register()`/`toProfile()` now touch `Membership`/`Organization` directly via Prisma — **`@pee/auth` deliberately never imports `@pee/organizations`** (see the circular-dependency note below).
- **`/services/organizations`** (`@pee/organizations`, **new in Phase 10**) — system of record for `Organization`/`Membership`, per `adr/0009`. `OrganizationsService` (`create`/`getOne`/`listForUser`/`listOrganizationIdsForUser`/`getDefaultOrganizationId`/`findMembership`/`assertRole`), `MembershipManagementService` (`listMembers`/`inviteMember`/`updateMemberRole`/`removeMember`), `OrganizationRolesGuard`+`@RequireRole()` decorator (reads the route's `:id` param as the target `organizationId`), `OrganizationsController` (`/organizations`), `MembersController` (`/organizations/:id/members`). Imports nothing from `@pee/auth` at the module level — its controllers use `JwtAuthGuard`/`CurrentUser` as plain imports, which resolve at runtime because Passport strategies register globally once `JwtStrategy` is instantiated anywhere in the app (confirmed: `@pee/projects` has done exactly this since Phase 2, with no `AuthModule` import). Owns `Organization`, `Membership`.
- **`/services/projects`** (`@pee/projects`) — first domain-entity module. `ProjectsController` (`/projects` CRUD, all routes `JwtAuthGuard`-protected), `ProjectsService` (Prisma access + org-membership enforcement). Owns `Project`. **Phase 10:** `findOwnedOrThrow` → `findAccessibleOrThrow` (checks `OrganizationsService.assertRole(userId, project.organizationId, 'MEMBER')` instead of strict `ownerId` equality — not-a-member still 404s); `create` resolves `organizationId` (from `dto.organizationId`, membership-checked, or defaults to the caller's personal org via `getDefaultOrganizationId`); `archive` additionally calls a new `assertDestructivePermission` (creator, or `assertRole(..., 'ADMIN')`) before archiving. Imports `OrganizationsModule`.
- **`/services/planning`** (`@pee/planning`) — second domain-entity module, nested two levels under `Project`. `goals/` (`GoalsService`, `ProjectGoalsController` at `/projects/:projectId/goals`, `GoalsController` at `/goals/:id`) and `tasks/` (`TasksService`, `GoalTasksController` at `/goals/:goalId/tasks`, `TasksController` at `/tasks/:id`), split into subfolders since the module owns two entities. Imports `ProjectsModule` and injects `ProjectsService` to verify project ownership on goal creation (reuses `getOne`, does not touch `ProjectsService`'s internals). `TasksService` injects `GoalsService` (same-module DI, not a cross-module reach-through) and calls `GoalsService.recalculateProgress(goalId)` after every task mutation — this is the "closed execution loop": task completion state propagates to the parent goal without a manual step. Owns `Goal`, `Task`. **Phase 10:** both services got the exact same `findAccessibleOrThrow`/`assertDestructivePermission` retrofit as `@pee/projects`; `create` inherits `organizationId` from the parent's already-fetched response (`ProjectsService.getOne`'s/`GoalsService.getOne`'s result now carries it) rather than taking it as a second Prisma call or caller input. Imports `OrganizationsModule` too.
- **`/services/execution`** (`@pee/execution`) — third domain module, but not a third owned entity in the Goal/Task hierarchy: it owns `TaskExecutionSession` (a real start/stop timer) and `ExecutionEvent` (an append-only status-change log). `events/` (`ExecutionEventsService` — `@OnEvent` listeners for `task.status_changed`/`goal.status_changed`, `listGoalActivity`, `listActiveSessions`; `GoalActivityController` at `/goals/:goalId/activity`) and `sessions/` (`TaskSessionsService` — `startTask`/`completeTask`, both reuse `TasksService.getOne`/`update`; `TaskSessionsController` at `/tasks/:taskId/execution`; `ActiveExecutionController` at `/execution/active`). Imports `PlanningModule` one-directionally (to call `TasksService`/`GoalsService`) — `@pee/planning` has no import of, or dependency on, `@pee/execution`; the two are connected only via `@nestjs/event-emitter`, avoiding a circular module dependency while still guaranteeing every status change is observed regardless of entry point. **Zero code changes in Phase 10** — `startTask`/`completeTask` call into the now-org-aware `TasksService.getOne`/`update`, so any org member can start/complete a teammate's task for free.
- **`/services/sync`** (`@pee/sync`) — implements `adr/0003`'s deferred sync protocol (Phase 5). `SyncEntityRegistryService` is the single registry of the three bidirectional, mutable, user-owned entities (`Project`/`Goal`/`Task`) — each entry pairs a Prisma delegate (for generic pull/optimistic-lock reads) with the entity's own domain-service `create`/`update` (for actual writes, so rollup/event side effects keep firing). `pull/sync-pull.service.ts` returns paginated (`take: 500`, `hasMore`) changes since a cursor. `push/sync-push.service.ts` applies each change: create-if-missing (passing through the client-generated id), or an atomic `updateMany`-with-version-guard when `clientVersion` matches, falling back to last-write-wins-by-timestamp when it doesn't. `SyncController` (`/sync/pull`, `/sync/push`) is `JwtAuthGuard`-protected like every other module. `ExecutionEvent` is deliberately **not** in the registry — see the "ExecutionEvent stays out of sync" decision below. **Zero code changes in Phase 10** — verified by reading `toChangeRecord`/`applyCreate`: neither includes or forwards `organizationId`, so an offline-created row synced from `apps/desktop`/`apps/mobile` silently lands in the caller's default personal org via `ProjectsService.create`'s existing default-resolution logic.
- **`/services/ai`** (`@pee/ai`) — implements `adr/0006`'s deferred `AIProvider` abstraction (Phase 6) plus its first real consumer. `provider/` — `AIProvider` interface (`complete()` only; `stream`/`embed` deliberately not built until a feature needs them), `AI_PROVIDER_TOKEN`-keyed DI factory (`createAIProvider`) that reads `AI_PROVIDER` and constructs only the active vendor's implementation — `AnthropicProvider` (forced tool-use for structured output) or `OpenAIProvider` (`response_format: json_schema`) — throwing at boot if that provider's API key is missing. Every vendor call runs under an explicit `AbortController` timeout (~20s), independent of SDK defaults. `recommendations/` — `AIRecommendationsService` (`generateTaskSuggestions`/`listRecommendations`/`accept`/`dismiss`), `AIRecommendationsController` (`/goals/:goalId/ai/task-suggestions`), `AIRecommendationActionsController` (`/ai/recommendations/:id/accept|dismiss`). Imports `PlanningModule` to reuse `GoalsService.getOne` (ownership) and `TasksService.create` (so accepted suggestions get the Phase 3 rollup, exactly like any other task creation). Owns `AIRecommendation`. **Zero code changes in Phase 10**, same reasoning as `@pee/execution`.
- **`/services/analytics`** (`@pee/analytics`) — read-only reporting layer over Phases 2-6's existing data (Phase 7). No new domain model owned. `SummaryService` (Project/Goal/Task status breakdowns via `groupBy`, total time tracked via `aggregate`, AI-recommendation acceptance rate), `VelocityService` (daily task/goal completion buckets over a capped trailing window, sourced from `ExecutionEvent`), `TimeTrackingService` (tracked-seconds totals grouped by goal or project, over a capped trailing window). `AnalyticsController` at `/analytics/summary|velocity|time-tracking`. Depends only on `@pee/database`/`@pee/auth`/`@pee/types` — deliberately no import of `@pee/planning`/`@pee/execution`/`@pee/ai`, since every query goes straight to `PrismaService` under the documented read-only-join carve-out (below), not through another module's public service. **Stays owner-scoped in Phase 10, a documented boundary not an oversight** — since it bypasses domain services by design, it inherits no org-scoping automatically, and widening it would be new work, not a retrofit. Tracked in `27-backlog.md`.
- **`/packages/local-client`** (`@pee/local-client`) — not a NestJS module; a plain TypeScript library proving the client half of the sync protocol. Owns a *separate* SQLite-backed Prisma schema (`prisma/schema.prisma`, generated to the gitignored `prisma/generated/client`, distinct from `@pee/database`'s Postgres client) mirroring `Project`/`Goal`/`Task`/`ExecutionEvent` plus two local-only bookkeeping tables (`SyncCursor`, `SyncOutboxEntry`). `LocalStore` performs offline-capable local writes (each one immediately durable, plus an outbox entry) and applies incoming pull changes. `SyncClient` calls `POST /sync/pull`/`POST /sync/push` over `fetch`, collapsing repeat local edits to the same row into one pushed change. **Consumed by `apps/desktop`** (Phase 8) directly, unmodified. **Not consumable by `apps/mobile`** (Phase 9) — Prisma's query engine has no Android/iOS binary target, so React Native ported the protocol instead (`adr/0008`); `@pee/local-client` is now the *reference implementation* both consumers are either built on or ported from.
- **`/apps/desktop`** (`@pee/desktop`) — not a NestJS module; an Electron app. Main process (`electron/main/`) runs in a Node.js context and imports `@pee/local-client`'s `LocalStore`/`SyncClient` directly (per `adr/0007`), exposing them to the sandboxed renderer only via typed IPC handlers (`ipc/projects-ipc.ts`/`goals-ipc.ts`/`tasks-ipc.ts` for local CRUD, `sync-ipc.ts` for manual+background sync, `remote-ipc.ts` for online-only passthroughs to `services/execution`/`services/ai`/`services/analytics`, `auth/auth-ipc.ts`+`auth/auth-session.ts` for login/token custody). `local-store-factory.ts` provisions a fresh local SQLite file under `app.getPath('userData')` on first launch via `prisma db push` (same approach as `@pee/local-client`'s own test fixture) — verified end-to-end in the authoring sandbox. The renderer (`src/`) is a new React+Vite app, not a reuse of `apps/web`'s Server Components (architecturally incompatible with an offline-first client needing local IPC reads) — it reuses `apps/web`'s Tailwind baseline/utility-class conventions instead. Depends on `@pee/local-client`/`@pee/types` only; no dependency on any `services/*` package — it talks to them exclusively over HTTP, exactly as `apps/web` does.
- **`/apps/mobile`** (`@pee/mobile`) — not a NestJS module; an Expo/React Native app. No main/renderer split (a single JS process), so its modules are consumed directly by screens rather than through an IPC boundary. `src/db/` — `schema.ts` (hand-written SQL mirroring `packages/local-client/prisma/schema.prisma` field-for-field), `connection.ts` (`expo-sqlite` bootstrap, idempotent schema creation on every launch), `mobile-store.ts` (facade composing `projects-repo.ts`/`goals-repo.ts`/`tasks-repo.ts`/`outbox-repo.ts` — same public method surface as `LocalStore`), `mobile-sync-client.ts` (a line-for-line port of `SyncClient`'s pull/push/conflict-resolution algorithm against `MobileStore`, `@pee/types` unmodified). `src/auth/` — `mobile-auth-session.ts` (mirrors `apps/desktop`'s `AuthSession`; `expo-secure-store` in place of `safeStorage`), `auth-context.tsx` (React Context, since there's no IPC boundary to carry session state across). `src/api/remote-client.ts` — online-only passthroughs to `services/execution`/`services/ai`/`services/analytics`, same scope boundary as Desktop. `src/sync/background-sync.ts` — manual + 30s-interval sync, mirrors `apps/desktop`'s `sync-ipc.ts` minus the IPC/window plumbing. Renderer (`src/screens`, `src/components`, `src/navigation`) uses NativeWind to reuse `apps/web`'s Tailwind conventions. Depends on `@pee/types` only; talks to every `services/*` package exclusively over HTTP. See `adr/0008-mobile-local-storage.md` for why `@pee/local-client` itself isn't imported here.
- **`/packages/database`** (`@pee/database`) — Prisma schema + `PrismaService`/`PrismaModule` (`@Global()`), shared by every Nest module needing DB access. Added beyond the originally enumerated package list (`sdk`, `ui`, `design-system`, `shared`, `config`, `types`, `utils`) because every future `/services` module needs DB access — recorded in `21-decision-log.md`.
- **`/packages/types`** (`@pee/types`) — shared request/response types (`RegisterRequest`, `LoginRequest`, `AuthTokens`, `UserProfile`, `CreateProjectRequest`, `UpdateProjectRequest`, `ProjectResponse`, `ListProjectsQuery`, `CreateGoalRequest`, `UpdateGoalRequest`, `GoalResponse`, `GoalProgress`, `ListGoalsQuery`, `CreateTaskRequest`, `UpdateTaskRequest`, `TaskResponse`, `ListTasksQuery`, `ExecutionEventResponse`, `TaskExecutionSessionResponse`, `ActiveExecutionResponse`, `SyncableEntityName`, `SyncChangeRecord`, `SyncPullResponse`, `SyncPushRequest`/`SyncPushResponse`, `PaginatedResponse<T>`, ...), consumed by every backend module, `@pee/local-client`, and the Next.js app. Also the shared home for the `task.status_changed`/`goal.status_changed` event-name constants and payload types, so `@pee/planning` (emitter) and `@pee/execution` (listener) agree on shape without importing each other, and (Phase 6) `AICompletionRequest`/`AICompletionResult`, `AIRecommendationResponse`/`AITaskSuggestion`, `AcceptRecommendationRequest`/`Response`, plus (Phase 7) `AnalyticsSummaryResponse`, `AnalyticsVelocityResponse`/`Point`, `AnalyticsTimeTrackingResponse`/`Entry`, plus (Phase 10) `organization.types.ts` (`MembershipRole`, `CreateOrganizationRequest`, `OrganizationResponse`, `MemberResponse`, `InviteMemberRequest`, `UpdateMemberRoleRequest`) and `sso.types.ts` (`SsoProvider`, `SsoStatusResponse`, `SsoProvisionRequest`/`Response` — the latter's `user` field is the full `UserProfile`, not a hand-picked subset, so Auth.js's `profile()` callback never needs a second round-trip).

## Implementation notes worth remembering

- Refresh tokens are **opaque random strings**, not JWTs — only their SHA-256 hash is persisted, so revocation is authoritative in the database rather than depending on JWT expiry. Rotation: each `/auth/refresh` call revokes the old token and issues a new one; replaying an already-revoked token revokes the user's entire active token chain (`TOKEN_REUSE_DETECTED`).
- The browser never receives a raw JWT — Auth.js (Next.js) holds tokens server-side; see [11-api-contract.md](11-api-contract.md) and [12-security.md](12-security.md).
- **Ownership pattern, reusable for future domain modules:** a resource module (e.g. `@pee/projects`, `@pee/planning`) never trusts a caller-supplied ID alone — it always re-checks `resource.ownerId === currentUser.id` in the service layer and returns 404 (not 403) on mismatch, so a caller can't distinguish "doesn't exist" from "exists but isn't yours."
- **Cross-module ownership checks go through the public service API, never raw Prisma:** when a resource nests under another module's entity (Goal under Project), the owning module's already-exported service method is reused (`ProjectsService.getOne`) rather than widening its private internals or reaching into its Prisma table directly. If no suitable public method existed, the fix would be adding one to the owning module — not reaching around it.
- **Soft delete pattern:** `DELETE` routes on domain entities set a `status`/`archivedAt`-or-`completedAt` field rather than removing the row — same convention `Project` uses, now also used by `Goal`/`Task`.
- **Closed-loop rollup pattern (Phase 3):** when a child entity's status feeds a parent entity's derived status/progress (Task → Goal), the rollup lives in the parent's service as a small public method (`recalculateProgress`) that the child's service calls after any mutation. Keeps the derivation logic in one place and makes the parent always consistent, without a background job.
- **Event-driven cross-module observability (new in Phase 4):** when a *sibling* module (not a parent/child in the same dependency direction) needs to observe every status change unconditionally, reach-through DI doesn't work without a cycle — `@pee/execution` needs to call *into* `@pee/planning`, but the logging hook has to live *inside* `@pee/planning` to see every code path. Resolved with `@nestjs/event-emitter`: `TasksService`/`GoalsService` `emit()` plain fact events with zero knowledge of who's listening; `@pee/execution` (or any future module) subscribes via `@OnEvent` without either module importing the other. Prefer this pattern over deeper reuse-chaining whenever the consuming module would otherwise need to import a module that itself needs to call back into the consumer.
- **No Prisma migration has been generated yet** — `prisma/migrations/` doesn't exist in the repo because no Docker/Postgres has been available in the authoring sandbox to run `prisma migrate dev` against. See [20-known-issues.md](20-known-issues.md).
- **`version` is now a live optimistic-concurrency guard, not a dead column (Phase 5):** every `update()`/`archive()` on `Project`/`Goal`/`Task` (and `GoalsService.recalculateProgress`) now does `version: { increment: 1 }` — `adr/0003` declared this column on every syncable table from Phase 1 onward, but no service ever incremented it until Phase 5 needed it for sync's optimistic lock. `SyncPushService`'s atomic guard is `updateMany({ where: { id, ownerId, version: expectedVersion }, data: { version: { increment: 1 } } })` — race-free by construction (no read-then-write gap), `count === 0` is the conflict signal.
- **Last-write-wins by wall-clock timestamp, not by version lineage:** when a push's `clientVersion` doesn't match the server's current version, the resolution compares `clientUpdatedAt` against the server row's `updatedAt`, not the version numbers — version is only ever the fast-path race guard, `updatedAt` is the actual source of truth for "whose edit is newer." This is why `Update*Dto`s gained an optional, sync-only `updatedAt?: Date` passthrough (never bound to the public HTTP DTO) — without it, the `@updatedAt` auto-stamp would overwrite an offline edit's true wall-clock time with "whenever it happened to sync," breaking the comparison.
- **Sync push reuses domain services for writes, not raw Prisma** (mirrors the Phase 4 "reuse over reimplement" discipline): creating/updating a row via sync calls `ProjectsService.create`/`update` etc., not `prisma.project.create` directly, specifically so a synced task completion still triggers the Phase 3 rollup and Phase 4 `ExecutionEvent` — sync must never be a silent bypass of those guarantees. The one exception is the atomic version-guard touch itself, which necessarily goes straight to Prisma (see above) — a two-step "guard then apply" purposely accepts a harmless double version-bump per applied write in exchange for keeping every existing service method's business logic untouched.
- **`id`/`updatedAt` overrides live on an internal `options` parameter, never on the public DTO:** `ProjectsService.create(ownerId, dto, options?)` etc. — `options.id`/`options.updatedAt` let the sync path pass through a client-generated UUID and the edit's true timestamp, but because they're a separate function argument (not a DTO field), no HTTP client can ever set them through the plain REST endpoints, even if `whitelist`/`forbidNonWhitelisted` were misconfigured. This is a stricter approach than the plan first sketched (which would have added the fields to the DTOs themselves).
- **`ExecutionEvent` deliberately stays out of the sync registry:** it's append-only and already served by Phase 4's `GET /goals/:goalId/activity`; adding a fourth, pull-only entity to a registry interface shaped around bidirectional entities would either force awkward no-op `applyCreate`/`applyUpdate` methods or a second registry shape. Descoped to backlog rather than forced in — see [27-backlog.md](27-backlog.md).

- **Single active AI provider, not automatic failover (Phase 6):** `createAIProvider` constructs only the provider named by `AI_PROVIDER` — the inactive vendor's key is never required. Rejected building a try-Claude-then-fall-back-to-OpenAI chain: it doubles cost on every degraded call and makes "which model produced this recommendation" non-deterministic, undermining §100's explainability requirement. A request-time failure returns a clean error for the client to retry, not a silent cross-vendor fallback.
- **Explainability is enforced at the recommendation boundary, not the raw-completion boundary:** `AIProvider.complete()` is a generic port — reason/confidence/alternatives don't universally apply to every possible completion (e.g. a future embedding call). `AIRecommendationResponse` (the shape every current/future recommendation feature returns) is what structurally requires those fields; `AIRecommendationsService` is the one place that boundary is enforced.
- **AI suggestions never auto-write to `Task` — human approval is a structural gate:** `generateTaskSuggestions` only persists an `AIRecommendation` row; nothing is written to `Task` until `accept` is called with specific indices. `accept` reuses `TasksService.create` (not raw Prisma), mirroring Phase 5's "sync push reuses domain services" discipline, so accepted suggestions get the same rollup/validation as any other task.
- **Never fabricate or partially return an AI result:** a provider throw or an unparseable structured response is caught, persisted as `AIRecommendation.status = FAILED` (for traceability of the failure itself), and surfaced as a clean `ServiceUnavailableException` — the same "fail gracefully" requirement from `evaluation/ai-feature-quality-bar.md`.

- **Analytics computes live, no materialized rollup (Phase 7):** `SummaryService`/`VelocityService`/`TimeTrackingService` all query at request time via Prisma `groupBy`/`aggregate`/bounded `findMany` — no summary table, no scheduled job. Per-user data volumes are small enough that this is simple and always-consistent; a rollup would be speculative infrastructure for a load that doesn't exist (Sustainable Complexity, Principle 8).
- **Analytics reads Prisma directly — the read-only-join carve-out extended to a whole module, not just one method (Phase 7):** Phase 4 established that a query already scoped by `ownerId`, with no authorization decision resting on the joined data, may use Prisma directly instead of round-tripping through another module's service (`listActiveSessions`). Phase 7 applies that same reasoning at the module level — `@pee/analytics` has no import of `@pee/planning`/`@pee/execution`/`@pee/ai` at all, since every one of its queries is exactly this shape.
- **Every analytics query is bounded by an explicit date-range cap:** `VelocityQueryDto.days` (1-90) and `TimeTrackingQueryDto.sinceDays` (1-365) prevent an unbounded full-history scan as an account ages — mirrors Phase 6's bounded-prompt precedent (cap cost at the point data is read).

- **Electron main process is the desktop BFF, not a new token-custody pattern (Phase 8):** `apps/desktop/electron/main/auth/auth-session.ts` holds the access token in memory and the refresh token encrypted at rest (`safeStorage`) — the renderer never sees either, only a `UserProfile`. This is the same "server holds the token" discipline `apps/web`'s Next.js server already applies, relocated to a different custody process, not a new pattern.
- **`@pee/local-client`'s sync scope (Project/Goal/Task only) is not widened for desktop:** execution sessions, AI suggestions, and analytics have no offline story on desktop — they're online-only IPC passthroughs to the same REST endpoints `apps/web` calls. Extending the sync registry to cover them is tracked in [27-backlog.md](27-backlog.md), not done implicitly.
- **IPC handlers derive `ownerId` from `AuthSession`, never from the renderer's call arguments (Phase 8):** every `projects-ipc.ts`/`goals-ipc.ts`/`tasks-ipc.ts` handler calls the shared `requireOwnerId(authSession)` helper — the same "never trust a caller-supplied identity" discipline as every HTTP controller's `ownerId`-from-JWT pattern, applied at the IPC boundary instead.

- **Prisma-based `LocalStore` cannot run on React Native — ported, not reused (Phase 9):** Prisma's query engine is a native binary with no published Android/iOS target, and Hermes/JSC is not a Node.js runtime, so `@pee/local-client` genuinely cannot be imported by `apps/mobile` the way `apps/desktop` imports it. `adr/0008` documents the resulting decision: a new `MobileStore` (expo-sqlite) with `LocalStore`'s exact method surface, and a `MobileSyncClient` that is a line-for-line port of `SyncClient`'s pull/push/conflict-resolution logic — `@pee/types` and the `services/sync` REST contract stay unmodified. This is the "or an equivalent" reading of the exit criteria, at the same abstraction boundary Phase 8 drew for "reuse the design system" once no design-system package existed.
- **`MobileStore` reads flow through explicit repo methods, not raw SQL scattered across callers (Phase 9):** `ProjectsRepo`/`GoalsRepo`/`TasksRepo` each expose `getById`/`listByX` used by both screens and `MobileSyncClient.buildPushChange` — a small, deliberate improvement over `@pee/local-client`'s own `SyncClient`, which reaches directly into `store.db.localProject.findUnique(...)` (Prisma-shape-coupled). The port fixes that coupling wart rather than reproducing it, without changing the algorithm's behavior.
- **Mobile auth/sync have no main/renderer split, so custody moves to React Context (Phase 9):** `apps/desktop`'s `AuthSession`/`SyncClient` are constructed once in the Electron main process and exposed via IPC; `apps/mobile` has one JS process, so `MobileAuthSession`/`BackgroundSyncRunner` are constructed once in `AuthProvider`/`SyncProvider` and exposed via React Context (`auth-context.tsx`/`sync-context.tsx`) instead — same "one place owns the token/sync state" discipline, adapted to a single-process shell.
- **`MobileStore`'s sync scope (Project/Goal/Task only) is not widened for mobile, same as desktop (Phase 9):** execution sessions, AI suggestions, and analytics stay online-only passthroughs (`api/remote-client.ts`) to the same REST endpoints `apps/web`/`apps/desktop` already call.

- **`@pee/auth` deliberately never imports `@pee/organizations` — a real circular-package-dependency bug, not a hypothetical one (Phase 10):** `@pee/organizations`'s controllers need `CurrentUser`/`JwtAuthGuard` from `@pee/auth`; a naive `AuthModule` importing `OrganizationsModule` (to inject `OrganizationsService` into `AuthService`) creates a genuine Node.js CommonJS `require()` cycle — confirmed by a runtime `TypeError: (0, auth_1.CurrentUser) is not a function` when actually tried, not caught by typechecking or `forwardRef()` (which only fixes Nest's DI-graph cycles, not the underlying module-loading order). Fixed by having `AuthService.toProfile()`/`createUserWithPersonalOrganization()` read `Membership`/`Organization` directly via Prisma — the same read-only-direct-Prisma carve-out `@pee/analytics` already uses (self-scoped read, no cross-user authorization decision rests on it) — and duplicating the ~3-line personal-org insert rather than calling `OrganizationsService`. See `adr/0009`. **Any future module that needs both `@pee/auth`'s guards and something `@pee/auth` itself needs to read is a candidate for this same pattern** — check for the cycle before adding the import, don't just add it and hope `forwardRef` covers it.
- **RBAC retrofit pattern, reusable for future domain modules (Phase 10):** each of `ProjectsService`/`GoalsService`/`TasksService` renamed its private `findOwnedOrThrow` → `findAccessibleOrThrow`, replacing a strict `ownerId` equality check with `OrganizationsService.assertRole(userId, resource.organizationId, 'MEMBER')` — same 404-for-non-member behavior, now membership-based instead of ownership-based. A second private helper, `assertDestructivePermission`, is called only from `archive()` and short-circuits for the creator (`resource.ownerId === userId`) before falling back to an `assertRole(..., 'ADMIN')` check. This is the template for any future destructive action that should stay creator-or-admin-gated while ordinary reads/updates stay member-gated.
- **`organizationId` is always inherited from the parent at creation time, never caller input, for `Goal`/`Task` (Phase 10):** `GoalsService.create` and `TasksService.create` already call `ProjectsService.getOne`/`GoalsService.getOne` respectively (to verify accessibility) — Phase 10 just reads `organizationId` off that same already-fetched response instead of making a second call or trusting a request field. `Project.create` is the one entry point that accepts an explicit (membership-checked) `organizationId`, since a project is the top of the hierarchy and has no parent to inherit from.
- **Sync/execution/AI needed zero code changes, verified by tracing, not assumed (Phase 10):** before writing anything, `services/sync/src/sync-entity-registry.service.ts`'s `toChangeRecord`/`applyCreate` were read line-by-line to confirm `organizationId` is never included in a pull payload and never forwarded on push — meaning offline-created rows from `apps/desktop`/`apps/mobile` land in the caller's default org for free, without touching either client. The same "does it actually delegate through the domain service" check was done for `@pee/execution`/`@pee/ai` before claiming they needed no changes.
- **SSO is additive to Auth.js, not a new auth system (Phase 10):** OIDC uses Auth.js's own native `type: 'oidc'` provider — this backend never implements an OIDC client; it only provisions a user after Auth.js has already verified the id_token itself (`POST /auth/sso/oidc/provision`, secret-header-guarded so only the trusted server-to-server caller can reach it). SAML has no native Auth.js provider type, so `services/auth/src/sso/saml/` implements a real SP (`@node-saml/node-saml`) behind a minimal OAuth2-authorization-code façade (`SamlBridgeService` — short-lived signed-JWT codes/tokens, not DB rows) so Auth.js's generic `type: 'oauth'` provider can drive it exactly like any OAuth IdP. Both funnel through one `SsoProvisioningService.findOrCreateUser`.
- **Security-critical validation logic gets its own file and its own direct unit test, not just controller-level coverage (Phase 10):** the SAML redirect allow-list check was extracted from a private controller method into `saml/redirect-allowlist.ts` specifically so a real open-redirect bypass (`redirectUri.startsWith(allowedOrigin)` letting `https://app.example.com.evil.com` through) could be caught by a dedicated test comparing parsed `URL` origins instead of string prefixes. Same treatment for `SsoProvisionGuard`'s secret comparison — `crypto.timingSafeEqual`, not `!==`, to avoid a timing side-channel.
