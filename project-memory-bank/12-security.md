# 12 — Security

Standards live in `claude/SECURITY.md` (source: `SYSTEM_PROMPT.md` §45). This file tracks the *product-specific* security posture.

## Status

**Phase 6 authz model unchanged, third-party AI data-sharing trade-off added, 2026-07-18** — `@pee/ai` reuses the same ownership/authentication pattern as every prior module; the new ground is sending user data to an external LLM vendor. See below for the AI-specific notes.

## Implemented model

- **Authentication:** email/password via the NestJS `auth` module (system of record) + Auth.js (NextAuth) on the Next.js frontend. Passwords hashed with argon2 (`PasswordService`). Access tokens are short-lived JWTs (default 15m); refresh tokens are opaque random strings — only their SHA-256 hash is persisted, so revocation is authoritative in the database, not dependent on JWT expiry.
- **Token custody (BFF pattern):** the browser never holds a raw access/refresh token. Auth.js's server-side session (encrypted httpOnly cookie) is the only thing the browser gets; the Next.js server calls the NestJS API directly. Removes CORS/token-exposure risk for the browser entirely at this phase.
- **Refresh rotation + reuse detection:** every `/auth/refresh` call revokes the presented token and issues a new one; presenting an already-revoked token revokes the user's *entire* active token chain and logs `TOKEN_REUSE_DETECTED` — standard theft-detection pattern.
- **Authorization:** a `role` field exists on `User` (`USER` today) as a least-privilege hook; not yet exercised by any endpoint (single role). **Resource-level authorization (Phase 2):** every `/projects/*` route requires `JwtAuthGuard`; `ProjectsService` additionally enforces ownership (`project.ownerId === currentUser.id`) before any read or write, throwing `NotFoundException` — not `ForbiddenException` — on mismatch, so a 404 response never confirms or denies that a given project ID exists for another user. This ownership-check pattern is the model for every future domain module. **Nested authorization (Phase 3):** `Goal` and `Task` each carry their own `ownerId` and are checked directly (same 404-not-403 rule) — `GoalsService`/`TasksService` don't need to walk up to the parent for a routine read/write. Goal *creation* is the one case that must check the parent first: `GoalsService.create` calls `ProjectsService.getOne(ownerId, projectId)` before creating a goal under it, so a caller can't create goals under a project they don't own. This crosses a module boundary through `ProjectsService`'s already-public API, not raw Prisma access — no internals of `@pee/projects` were changed to make this work.
- **Rate limiting:** `@nestjs/throttler`, global default (100/min) plus tighter limits on `/auth/register` (5/min) and `/auth/login` (10/min). `/projects/*` and `/goals/*`/`/tasks/*` rely on the global default only — CRUD on owned resources isn't a credential-guessing surface, so no tighter per-route limit was added.
- **Audit logging:** `AuthAuditLog` records `LOGIN_SUCCESS`, `LOGIN_FAILURE`, `LOGOUT`, `TOKEN_REFRESH`, `TOKEN_REUSE_DETECTED`. **Partially closed in Phase 4:** `ExecutionEvent` (owned by `@pee/execution`) now audit-logs every `Goal`/`Task` status transition, append-only, fired unconditionally via `@nestjs/event-emitter` regardless of which endpoint caused the change — a caller cannot bypass it by using the generic `PATCH` route instead of the dedicated start/complete endpoints. **Remaining gap:** create/rename/description-edit actions on `Project`/`Goal`/`Task` (non-status-changing) are still not audit-logged — `ExecutionEvent` only covers status transitions, not general CRUD. See [20-known-issues.md](20-known-issues.md) and [27-backlog.md](27-backlog.md).
- **Cross-module observability without a cyclic dependency (Phase 4):** `@pee/planning`'s `TasksService`/`GoalsService` emit plain fact events (`task.status_changed`, `goal.status_changed`) via `EventEmitter2` — they have no import of, or knowledge of, `@pee/execution`. `@pee/execution` listens (`@OnEvent`) and separately depends on `PlanningModule` one-directionally to call `TasksService.update`/`getOne` for its own start/complete logic. This guarantees every status change is logged regardless of entry point, without planning ever depending on execution.
- **Read-path carve-out (Phase 4):** `ExecutionEventsService.listActiveSessions` reads `Task.title`/`Goal.title` via a Prisma `include` on the caller's own `TaskExecutionSession` rows, rather than round-tripping through `TasksService`/`GoalsService` for each one. This is safe because the session row is already scoped to the caller via `TaskExecutionSession.ownerId` — no authorization decision is made from the joined data, only display text. The "cross-module reads go through the public service API" rule (Phase 3) exists to protect ownership *decisions*, not arbitrary read joins for display purposes.
- **Sync push never trusts client-supplied ownership (Phase 5):** `SyncPushService` forces `ownerId` from the authenticated caller (`CurrentUser`) on every change, exactly like every other write path; a push targeting an `id` that exists but belongs to another owner is rejected with `reason: 'not_found'` — the same 404-shaped non-disclosure as every other cross-owner access, never a 403 that would confirm the row exists. The optimistic-lock `updateMany` guard additionally re-scopes by `ownerId` at the database level as defense in depth, independent of the earlier application-level check.
- **Per-entity payload validation on push (Phase 5):** the outer `SyncPushRequestDto` validates shape only (`entity` is one of the three known names, `id` is a UUID, `clientVersion` is a positive int); the opaque `data` blob is then validated a second time, programmatically, against the matching entity's own `*SyncFieldsDto` (`whitelist: true, forbidNonWhitelisted: true`) before any write — a client can't smuggle an unexpected field (e.g. `ownerId`) through the generic `data` object just because the outer DTO doesn't type-check it.
- **Third-party data sharing, new ground in Phase 6:** `POST /goals/:goalId/ai/task-suggestions` sends the goal's title/description and up to 50 existing task titles to a third-party LLM vendor (Anthropic or OpenAI, whichever `AI_PROVIDER` selects) — the first time any request in this product sends user data outside the server boundary. Accepted trade-off: it's the user's own data, sent to help that same user plan their own work; no new PII is introduced beyond what the user already typed into the product. Documented explicitly here rather than left implicit.
- **AI generate endpoint has a tighter rate limit than the global default (Phase 6):** `@Throttle({ default: { limit: 10, ttl: 60_000 } })` on `POST /goals/:goalId/ai/task-suggestions`, versus the global 100/min — LLM calls are slow and cost real money per call, unlike ordinary CRUD on an owned resource.
- **AI provider calls are bounded and fail closed, never fabricate (Phase 6):** every vendor call runs under an explicit ~20s `AbortController` timeout, independent of SDK defaults, so a hung vendor call can't tie up a request indefinitely. A provider error or unparseable structured output is caught, the attempt persisted as `AIRecommendation.status = FAILED` (for traceability), and the client gets a generic `ServiceUnavailableException` — the raw vendor error (which could contain request/response internals) never reaches the response body.
- **AI recommendations never bypass ownership or write without human approval (Phase 6):** `AIRecommendationsService` forces `ownerId` from the JWT on every call (never trusts the payload), re-verifies goal ownership via `GoalsService.getOne` (404-not-403), and never writes to `Task` except via the explicit `accept` action for the specific suggestions the caller selected — the same reuse-domain-services discipline as Phase 5's sync push.
- **AI provider API keys are fail-fast, not fail-silent (Phase 6):** the active provider's key (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`, selected by `AI_PROVIDER`) is required at application boot — a misconfigured deployment fails immediately and visibly, rather than accepting requests it can't actually serve.
- **Secrets:** `.env` (never committed; `.env.example` documents required keys), read via `@nestjs/config`.
- **Transport security:** `helmet()` sets standard security headers; TLS termination is a deployment-time requirement (`adr/0004`), not application code.
- **Local SQLite file is unencrypted at rest (Phase 5, known gap):** `packages/local-client`'s SQLite file holds a plain-text copy of the caller's `Project`/`Goal`/`Task` data on whatever device runs it. No worse than the Postgres side (disk encryption is an infra concern there too), but a device-loss scenario is a more realistic threat model for a local file than for a managed database. Tracked in [27-backlog.md](27-backlog.md) — revisit alongside Phase 8 (Desktop), where this file will actually ship on end-user machines.
- **No sensitive data in logs:** only `tokenHash` is ever persisted for refresh tokens; passwords and raw tokens never appear in any log statement.
- **Dependency scan:** `npm audit` run 2026-07-17 — remaining findings (Next.js, NestJS/Express-chain transitive advisories) all require a major-version bump (Next 16, Nest 11) to clear; deliberately not force-upgraded mid-feature. Tracked in [20-known-issues.md](20-known-issues.md).
- **Compliance:** none identified yet.

Do not duplicate the general security requirements checklist here — it lives in `claude/SECURITY.md` and `checklists/security-checklist.md`.
