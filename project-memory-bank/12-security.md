# 12 — Security

Standards live in `claude/SECURITY.md` (source: `SYSTEM_PROMPT.md` §45). This file tracks the *product-specific* security posture.

## Status

**Phase 10 adds the first new server-side authorization model since Phase 2 (multi-tenancy/RBAC) and the first federated-identity surface (OIDC/SAML SSO), 2026-07-18.** Two real security issues were found in this phase's own review pass and fixed before shipping (non-constant-time secret comparison; a SAML redirect open-redirect bypass) — both documented below with their fixes and regression tests. A genuine circular-package-dependency bug (`@pee/auth` ⇄ `@pee/organizations`) was also found and fixed — see [08-backend-guidelines.md](08-backend-guidelines.md) and `adr/0009`.

## Implemented model

- **Authentication:** email/password via the NestJS `auth` module (system of record) + Auth.js (NextAuth) on the Next.js frontend. Passwords hashed with argon2 (`PasswordService`). Access tokens are short-lived JWTs (default 15m); refresh tokens are opaque random strings — only their SHA-256 hash is persisted, so revocation is authoritative in the database, not dependent on JWT expiry.
- **Token custody (BFF pattern):** the browser never holds a raw access/refresh token. Auth.js's server-side session (encrypted httpOnly cookie) is the only thing the browser gets; the Next.js server calls the NestJS API directly. Removes CORS/token-exposure risk for the browser entirely at this phase.
- **Refresh rotation + reuse detection:** every `/auth/refresh` call revokes the presented token and issues a new one; presenting an already-revoked token revokes the user's *entire* active token chain and logs `TOKEN_REUSE_DETECTED` — standard theft-detection pattern.
- **Authorization:** a `role` field exists on `User` (`USER` today) as a least-privilege hook; not yet exercised by any endpoint (single role — this is a *global* role, unrelated to the *per-organization* `MembershipRole` added in Phase 10, below). **Resource-level authorization (Phase 2):** every `/projects/*` route requires `JwtAuthGuard`; `ProjectsService` additionally enforces ownership before any read or write, throwing `NotFoundException` — not `ForbiddenException` — on mismatch. **Nested authorization (Phase 3):** `Goal` and `Task` each carry their own `ownerId` and are checked directly (same 404-not-403 rule). **Retrofitted to membership-based authorization in Phase 10** — see below; the underlying 404-not-403 discipline is unchanged, only what's being checked (org membership, not strict ownership) is different.
- **Multi-tenancy and RBAC (Phase 10):** new `Organization`/`Membership` model (`MembershipRole`: `OWNER`/`ADMIN`/`MEMBER`) — every user gets an invisible personal org + `OWNER` membership at registration. `ProjectsService`/`GoalsService`/`TasksService`'s `findOwnedOrThrow` → `findAccessibleOrThrow`: not-a-member of the resource's organization is still 404 (existence-hiding, unchanged discipline); a member with insufficient role is **403** — a deliberate, narrow exception to the blanket 404-not-403 rule, since being a member of an org you can already see doesn't newly disclose anything (see `adr/0009`). Any `MEMBER`+ can read/create/update; only the creator (`ownerId === caller`) or an `ADMIN`/`OWNER` can archive/delete (`assertDestructivePermission`). `services/execution`/`services/ai`/`services/sync` inherited this for free (they delegate writes through the retrofitted services) — verified by tracing, not assumed. `services/analytics` deliberately stays owner-scoped (it bypasses domain services by design) — a documented boundary, tracked in `27-backlog.md` if ever widened.
- **A genuine circular package dependency, found and fixed (Phase 10):** `@pee/organizations`'s controllers need `CurrentUser`/`JwtAuthGuard` from `@pee/auth`; a first attempt at also having `@pee/auth` import `OrganizationsService` from `@pee/organizations` (to build a personal org at registration and read it back for `UserProfile.organizations`) created a real Node.js `require()` cycle — confirmed by a runtime `TypeError`, not merely a lint warning. Fixed by having `AuthService` read `Membership`/`Organization` directly via Prisma (the same read-only-direct-Prisma carve-out `@pee/analytics` uses) instead of calling into `@pee/organizations` at all.
- **Rate limiting:** `@nestjs/throttler`, global default (100/min) plus tighter limits on `/auth/register` (5/min) and `/auth/login` (10/min). `/projects/*` and `/goals/*`/`/tasks/*` rely on the global default only — CRUD on owned resources isn't a credential-guessing surface, so no tighter per-route limit was added.
- **Audit logging:** `AuthAuditLog` records `LOGIN_SUCCESS`, `LOGIN_FAILURE`, `LOGOUT`, `TOKEN_REFRESH`, `TOKEN_REUSE_DETECTED`. **Partially closed in Phase 4:** `ExecutionEvent` (owned by `@pee/execution`) now audit-logs every `Goal`/`Task` status transition, append-only, fired unconditionally via `@nestjs/event-emitter` regardless of which endpoint caused the change — a caller cannot bypass it by using the generic `PATCH` route instead of the dedicated start/complete endpoints. **Remaining gap:** create/rename/description-edit actions on `Project`/`Goal`/`Task` (non-status-changing) are still not audit-logged — `ExecutionEvent` only covers status transitions, not general CRUD. See [20-known-issues.md](20-known-issues.md) and [27-backlog.md](27-backlog.md).
- **Cross-module observability without a cyclic dependency (Phase 4):** `@pee/planning`'s `TasksService`/`GoalsService` emit plain fact events (`task.status_changed`, `goal.status_changed`) via `EventEmitter2` — they have no import of, or knowledge of, `@pee/execution`. `@pee/execution` listens (`@OnEvent`) and separately depends on `PlanningModule` one-directionally to call `TasksService.update`/`getOne` for its own start/complete logic. This guarantees every status change is logged regardless of entry point, without planning ever depending on execution.
- **Read-path carve-out (Phase 4):** `ExecutionEventsService.listActiveSessions` reads `Task.title`/`Goal.title` via a Prisma `include` on the caller's own `TaskExecutionSession` rows, rather than round-tripping through `TasksService`/`GoalsService` for each one. This is safe because the session row is already scoped to the caller via `TaskExecutionSession.ownerId` — no authorization decision is made from the joined data, only display text. The "cross-module reads go through the public service API" rule (Phase 3) exists to protect ownership *decisions*, not arbitrary read joins for display purposes.
- **Sync push never trusts client-supplied ownership (Phase 5):** `SyncPushService` forces `ownerId` from the authenticated caller (`CurrentUser`) on every change, exactly like every other write path; a push targeting an `id` that exists but belongs to another owner is rejected with `reason: 'not_found'` — the same 404-shaped non-disclosure as every other cross-owner access, never a 403 that would confirm the row exists. The optimistic-lock `updateMany` guard additionally re-scopes by `ownerId` at the database level as defense in depth, independent of the earlier application-level check.
- **Per-entity payload validation on push (Phase 5):** the outer `SyncPushRequestDto` validates shape only (`entity` is one of the three known names, `id` is a UUID, `clientVersion` is a positive int); the opaque `data` blob is then validated a second time, programmatically, against the matching entity's own `*SyncFieldsDto` (`whitelist: true, forbidNonWhitelisted: true`) before any write — a client can't smuggle an unexpected field (e.g. `ownerId`) through the generic `data` object just because the outer DTO doesn't type-check it.
- **Third-party data sharing, new ground in Phase 6:** `POST /goals/:goalId/ai/task-suggestions` sends the goal's title/description and up to 50 existing task titles to a third-party LLM vendor (Anthropic or OpenAI, whichever `AI_PROVIDER` selects) — the first time any request in this product sends user data outside the server boundary. Accepted trade-off: it's the user's own data, sent to help that same user plan their own work; no new PII is introduced beyond what the user already typed into the product. Documented explicitly here rather than left implicit.
- **AI generate endpoint has a tighter rate limit than the global default (Phase 6):** `@Throttle({ default: { limit: 10, ttl: 60_000 } })` on `POST /goals/:goalId/ai/task-suggestions`, versus the global 100/min — LLM calls are slow and cost real money per call, unlike ordinary CRUD on an owned resource.
- **AI provider calls are bounded and fail closed, never fabricate (Phase 6):** every vendor call runs under an explicit ~20s `AbortController` timeout, independent of SDK defaults, so a hung vendor call can't tie up a request indefinitely. A provider error or unparseable structured output is caught, the attempt persisted as `AIRecommendation.status = FAILED` (for traceability), and the client gets a generic `ServiceUnavailableException` — the raw vendor error (which could contain request/response internals) never reaches the response body.
- **AI recommendations never bypass ownership or write without human approval (Phase 6):** `AIRecommendationsService` forces `ownerId` from the JWT on every call (never trusts the payload), re-verifies goal ownership via `GoalsService.getOne` (404-not-403), and never writes to `Task` except via the explicit `accept` action for the specific suggestions the caller selected — the same reuse-domain-services discipline as Phase 5's sync push.
- **AI provider API keys are fail-fast, not fail-silent (Phase 6):** the active provider's key (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`, selected by `AI_PROVIDER`) is required at application boot — a misconfigured deployment fails immediately and visibly, rather than accepting requests it can't actually serve.
- **Analytics introduces no new data exposure (Phase 7):** every `/analytics/*` route is `JwtAuthGuard`-protected and every service method takes `ownerId` as an explicit parameter sourced from the JWT — there is no cross-owner read path to even attempt. Confirmed by an e2e test seeding two owners' data and asserting neither's aggregates include the other's. The read-only-direct-Prisma pattern (bypassing `@pee/planning`/`@pee/execution`/`@pee/ai`) is safe under the same carve-out as Phase 4's `listActiveSessions`: the query is already owner-scoped before any join runs, and no authorization decision is made from the joined data.
- **Analytics query parameters are capped, closing a DoS-shaped surface (Phase 7):** `VelocityQueryDto.days` (1-90) and `TimeTrackingQueryDto.sinceDays` (1-365) are validated server-side (`class-validator`, global `ValidationPipe` with `whitelist`/`forbidNonWhitelisted`) — a caller cannot request an unbounded date range to force an expensive full-history scan.
- **Secrets:** `.env` (never committed; `.env.example` documents required keys), read via `@nestjs/config`.
- **Transport security:** `helmet()` sets standard security headers; TLS termination is a deployment-time requirement (`adr/0004`), not application code.
- **Local SQLite file is unencrypted at rest (Phase 5 gap, now live rather than hypothetical — Phase 8):** `packages/local-client`'s SQLite file holds a plain-text copy of the caller's `Project`/`Goal`/`Task` data. Phase 5 flagged this as "revisit before Phase 8 ships this file to end-user machines" — Phase 8 now does exactly that (`apps/desktop` provisions this file under `app.getPath('userData')`). Still not encrypted; a device-loss scenario is now a genuinely live threat, not a future one. Tracked in [27-backlog.md](27-backlog.md) with elevated priority.
- **Electron hardening baseline applied, not assumed (Phase 8):** `apps/desktop`'s `BrowserWindow` sets `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true`; the renderer's only channel to the main process is a narrow, typed `contextBridge` surface (`electron/preload/index.ts`) — no raw `ipcRenderer` is ever exposed to renderer code. `index.html` sets a `script-src 'self'` CSP. No remote content is ever loaded into the `BrowserWindow` (local dev server URL or a bundled `index.html` only).
- **Desktop token custody mirrors the BFF pattern, relocated (Phase 8):** `AuthSession` (Electron main process) holds the access token in memory only and persists the refresh token via OS-keychain-backed `safeStorage.encryptString`, never plain JSON — the renderer receives only a `UserProfile`, never a token, identical in spirit to Auth.js's server-side session on `apps/web`. **Known degrade:** if `safeStorage.isEncryptionAvailable()` is false (e.g. some Linux configurations without a keyring), the refresh token falls back to a plain-text file on disk — documented, not silently accepted as equivalent.
- **IPC payload shape is not validated with `class-validator` the way every HTTP DTO is (Phase 8, known gap):** `projects-ipc.ts`/`goals-ipc.ts`/`tasks-ipc.ts` pass renderer-supplied fields straight to `LocalStore`. The trust boundary differs from a public HTTP endpoint (the renderer runs code this project authored, not arbitrary internet input), but a compromised renderer (e.g. a future supply-chain issue in a renderer dependency) is a realistic desktop threat model this doesn't yet defend against. Tracked in [27-backlog.md](27-backlog.md).
- **Desktop's online-only surfaces reuse `services/api`'s existing rate limits, unmodified (Phase 8):** `remote-ipc.ts`'s passthroughs to execution/AI/analytics hit the same endpoints, same throttler config, as `apps/web` — no new server-side surface, no new limit needed.
- **A second unencrypted local SQLite file now ships, on a second device class (Phase 9):** `apps/mobile`'s `MobileStore` provisions its own on-device SQLite file via `expo-sqlite`, holding the same plain-text `Project`/`Goal`/`Task` data as `apps/desktop`'s file. Same known gap as Phase 8, now live on phones/tablets too — a lost/stolen mobile device is arguably a more common real-world scenario than a lost desktop. Tracked in [27-backlog.md](27-backlog.md) with the same elevated priority as Desktop's.
- **Mobile token custody mirrors the BFF pattern via the OS keychain, not a new pattern (Phase 9):** `MobileAuthSession` holds the access token in memory only and persists the refresh token via `expo-secure-store` (iOS Keychain/Android Keystore-backed) — screens receive only a `UserProfile` via `useAuth()`, never a token. Same discipline as Desktop's `safeStorage`, adapted to a single-process shell with no IPC boundary.
- **Mobile's `ownerId` is always derived from the authenticated session, never from a screen's local state (Phase 9):** every `MobileStore.createProject`/`createGoal`/`createTask` call in a screen passes `user.id` sourced from `useAuth()`, the same "never trust a caller-supplied identity" discipline as Desktop's `requireOwnerId(authSession)` helper, applied without an IPC boundary to enforce it at.
- **Mobile's online-only surfaces reuse `services/api`'s existing rate limits, unmodified (Phase 9):** `api/remote-client.ts`'s passthroughs to execution/AI/analytics hit the same endpoints, same throttler config, as `apps/web`/`apps/desktop` — no new server-side surface, no new limit needed.
- **No sensitive data in logs:** only `tokenHash` is ever persisted for refresh tokens; passwords and raw tokens never appear in any log statement.
- **SSO provisioning endpoints are secret-guarded, server-to-server only (Phase 10):** `POST /auth/sso/oidc/provision` and the SAML bridge's `token`/`userinfo` endpoints must never be reachable by the browser directly — without a positive control, a caller could mint an `SsoIdentity` for an arbitrary email/providerUserId and get issued real tokens. `SsoProvisionGuard` checks a shared secret (`SSO_INTERNAL_SECRET`) via `crypto.timingSafeEqual`, **not** a plain `!==` — a naive string comparison would let an attacker infer the secret byte-by-byte via response-timing differences. Found in this phase's own review pass, fixed, and covered by a dedicated test before merge, not left as a latent bug.
- **SAML redirect allow-list compares parsed origins, not a string prefix (Phase 10):** the SAML bridge's `authorize`/`acs` endpoints validate the caller-supplied `redirect_uri` against `SSO_SAML_ALLOWED_REDIRECT_ORIGIN`. An earlier version used `redirectUri.startsWith(allowedOrigin)`, which is a classic open-redirect bypass — `https://app.example.com.evil.com/...` also "starts with" `https://app.example.com`. Fixed by parsing both URLs and comparing `.origin` exactly (`saml/redirect-allowlist.ts`), with a dedicated unit test asserting the exact bypass string is rejected. Also fail-closed if the allow-list itself isn't configured (503, not "allow everything").
- **SAML SP validation fails closed on any parsing/signature/timing error, never fabricates an identity (Phase 10):** `SamlService.validateResponse` wraps the entire `@node-saml/node-saml` `validatePostResponseAsync` call in a try/catch mapping any failure to a generic 401 — the raw library error (which could contain XML internals) never reaches a response body, mirroring Phase 6's "vendor errors are mapped before they reach a response" rule for AI provider calls. Genuinely tested against garbage input and a well-formed-but-unsigned SAML response, both rejected — using the real `@node-saml/node-saml` library, not a mock.
- **SSO is feature-flagged off by default, fail-closed (Phase 10):** both OIDC and SAML require their respective env vars (`SSO_OIDC_ISSUER_URL`, `SSO_SAML_ENTRY_POINT`, etc.); missing config means the routes return a clean 503, not a silent no-op or a misleadingly-successful response. `GET /auth/sso/status` is how `apps/web` learns which buttons to show, mirroring Phase 6's AI-provider-key fail-fast posture.
- **SAML bridge secret no longer falls back across trust domains (P0 hardening, 2026-07-19):** `SamlBridgeService.secret()` previously fell back from `SSO_SAML_BRIDGE_SECRET` to `JWT_ACCESS_SECRET`, then to `''`, if unset — a real secret-reuse-across-trust-domains bug (a bug in either domain's secret handling could leak into the other) and a worst case of signing with an empty key. Found by an independent production-hardening audit, fixed the same session: the method now throws `ServiceUnavailableException` (503) with **no fallback of any kind**. Regression test in `services/auth/test/sso/saml/saml-bridge.service.spec.ts` asserts `JWT_ACCESS_SECRET` is never even read when `SSO_SAML_BRIDGE_SECRET` is missing.
- **All 11 `SSO_*` env vars actually referenced in code are now documented (P0 hardening, 2026-07-19):** `.env.example` previously documented zero of them; a real deployment following the example file could misconfigure SSO or hit the secret-reuse bug above with no warning. Now all of `SSO_INTERNAL_SECRET`/`SSO_OIDC_ISSUER_URL`/`SSO_OIDC_CLIENT_ID`/`SSO_OIDC_CLIENT_SECRET`/`SSO_SAML_ENTRY_POINT`/`SSO_SAML_ISSUER`/`SSO_SAML_IDP_CERT`/`SSO_SAML_CALLBACK_URL`/`SSO_SAML_CLIENT_ID`/`SSO_SAML_BRIDGE_SECRET`/`SSO_SAML_ALLOWED_REDIRECT_ORIGIN` are listed with comments explaining what enables what.
- **Boot-time env validation (P0 hardening, 2026-07-19):** `services/api/src/env.validation.ts` — a Joi schema on `ConfigModule.forRoot` — requires `DATABASE_URL`/`JWT_ACCESS_SECRET` unconditionally, the active `AI_PROVIDER`'s API key conditionally, and every `SSO_SAML_*` var conditionally once `SSO_SAML_ENTRY_POINT` is set (a partially-configured SAML provider is worse than a cleanly-disabled one). A misconfigured deployment now fails loudly at startup instead of confusingly on the first request that touches the missing config — same fail-fast posture Phase 6 already applied to the AI provider key, now generalized.
- **`/health` is a real readiness probe, not a hardcoded response (P0 hardening, 2026-07-19):** previously returned `{status:'ok'}` unconditionally, so an orchestrator load-balancing on it could route traffic to an instance that couldn't reach its database. Now runs a real Prisma `SELECT 1` and returns 503 if unreachable. `main.ts` also now calls `app.enableShutdownHooks()`, so `SIGTERM` drains in-flight requests and disconnects Prisma cleanly instead of dropping connections mid-query.
- **Structured, correlatable logs exist for the first time (P0 hardening, 2026-07-19):** every backend log line was previously an ad hoc `console.log`/nothing at all — there was no way to answer "what happened" for a given request. New `@pee/logging` package: JSON-line logger, `AsyncLocalStorage`-based request/user-id correlation, applied globally in `services/api`. No sensitive data (tokens, password hashes) is included in any log line — same discipline as the pre-existing "no sensitive data in logs" rule above, just now with actual logs to apply it to.
- **An SSO-only user has no password at all, not a placeholder hash (Phase 10):** `User.passwordHash` is nullable; `AuthService.login()` explicitly checks `!user.passwordHash` before ever calling `PasswordService.verify` — an SSO-provisioned account genuinely cannot be logged into with a password, not merely "the password is unguessable."
- **Known, accepted gap — invite-by-email can reveal account existence (Phase 10):** `MembershipManagementService.inviteMember` returns a different error for "no account with that email" vs. other failures, which an authenticated `ADMIN`/`OWNER` could use to probe whether an email has a PEE account. Lower-risk than a public registration form (requires already being an org admin), but a real, documented trade-off — not silently accepted. Tracked in [27-backlog.md](27-backlog.md) alongside "email-token-based invites," which would remove the need for this lookup entirely.
- **Dependency scan:** `npm audit` run 2026-07-17 — remaining findings (Next.js, NestJS/Express-chain transitive advisories) all require a major-version bump (Next 16, Nest 11) to clear; deliberately not force-upgraded mid-feature. Re-run 2026-07-18 after adding `apps/mobile`: 24 additional findings, all inside Expo's own CLI/prebuild tooling — deliberately deferred. Phase 10 added `@node-saml/node-saml` (and its transitive `xml-crypto`/`xml-encryption`/`xml2js` dependencies) — no new findings introduced by it as of this scan.
- **Compliance:** none identified yet.
