## Architecture

**Resolved 2026-07-17 (`adr/0002`-`adr/0006`); Phase 1 (Authentication) implemented against it same day, Phases 2-7 (Projects, Planning Engine, Execution Engine, Memory Engine, AI Integration, Analytics) implemented 2026-07-18, Phase 8 (Desktop) implemented 2026-07-18, Phase 9 (Mobile) implemented 2026-07-18.** Backend: TypeScript/NestJS modular monolith. Storage: PostgreSQL (implemented) + SQLite local (implemented, `packages/local-client`, consumed unmodified by `apps/desktop`, and ported to `expo-sqlite` for `apps/mobile` since Prisma cannot run on React Native). Infra: Docker/docker-compose (implemented for dev/CI), GitHub Actions (implemented), K8s/Terraform deferred. Auth: first-party NestJS module + Auth.js, JWT sessions — implemented; Phase 8 relocated the same token-custody discipline into Electron's main process, Phase 9 relocated it again into a React Context-held `MobileAuthSession` backed by `expo-secure-store`. AI: first-party provider interface, Claude + OpenAI — implemented, consumed by a real feature (goal → task-breakdown suggestions), now also reachable from both the desktop and mobile clients as an online-only passthrough. Analytics: no dedicated ADR exists — Phase 7 introduced a plain query/reporting layer (live aggregation, no new architectural pattern), documented as a decision-log entry rather than a new ADR since it doesn't change any structural decision the ADRs govern. Desktop shell: `adr/0007` — Electron, chosen specifically because its Node.js main process can run `@pee/local-client` unmodified, satisfying Phase 8's "no rewrite" exit criterion. Mobile local storage: `adr/0008` — a ported `expo-sqlite` storage engine + a literal port of `SyncClient`'s algorithm, chosen because Prisma's query engine has no Android/iOS binary target, making literal reuse technically impossible; the exit criteria's "or an equivalent" clause is exercised here. Full detail and rationale: [03-system-architecture.md](03-system-architecture.md) and `adr/0002`-`adr/0008`.
