# 20 — Known Issues / Risk Register

Source of truth for process: `SYSTEM_PROMPT.md` §82 (`System_Prompt/Part5.md`).

## Open risks

| Risk | Type | Impact | Mitigation |
|---|---|---|---|
| `apps/web` itself has no offline capability | Technical | Local-First (Principle 2) isn't realized in the shipped web product yet, even though the sync protocol now exists and Phase 8 proved it out for desktop | Phase 5 built the protocol (`services/sync`) and a reusable reference client (`packages/local-client`); Phase 8 consumed it for `apps/desktop`, but `apps/web` itself (100% server-rendered, no client-side storage) remains unretrofitted — Phase 9 (Mobile) is next to consume it, `apps/web` retrofit still not scheduled; see [27-backlog.md](27-backlog.md) |
| `ExecutionEvent` is not covered by the sync protocol | Technical | Neither a future offline web UI nor `apps/desktop` can show the activity timeline while disconnected | Deliberately descoped from Phase 5's registry (bidirectional-only, 3 entities) rather than forcing an awkward pull-only shape into it; already served online by `GET /goals/:goalId/activity` (and by `apps/desktop`'s online-only execution passthrough). Revisit if/when an offline activity view is actually needed — see [27-backlog.md](27-backlog.md) |
| `packages/local-client`'s SQLite file is unencrypted at rest, and now genuinely ships (Phase 8) | Security | A lost/stolen device running `apps/desktop` exposes plain-text `Project`/`Goal`/`Task` data — Phase 5 flagged this as "revisit before Phase 8 ships this file to real end-user machines," and Phase 8 now does exactly that | No worse than the current Postgres-side posture (disk encryption is an infra concern there too), but this is no longer a future concern — `apps/desktop` provisions this exact file under `app.getPath('userData')` today. Elevated priority in [27-backlog.md](27-backlog.md) |
| `apps/desktop`'s IPC handlers don't validate payload shape with `class-validator` the way every HTTP DTO does | Security | A compromised renderer (e.g. a future supply-chain issue in a renderer dependency) could send malformed IPC payloads straight to `LocalStore`, unlike the public HTTP API's `whitelist`/`forbidNonWhitelisted` validation | The renderer is code this project authored (not arbitrary internet input) and is sandboxed via `contextIsolation`/`sandbox`, so the risk is lower than a public endpoint, but not zero. Add `class-validator` DTOs to the IPC boundary as a follow-up — see [27-backlog.md](27-backlog.md) |
| `apps/desktop` has no code signing, auto-update, or cross-platform CI packaging matrix | Technical/Release | `electron-builder.yml` produces an unsigned local build only; no distribution pipeline exists | Deliberately scoped out of Phase 8 — signing certs and a packaging matrix are a distribution concern, not a "does the app work" concern. See [27-backlog.md](27-backlog.md) |
| SYSTEM_PROMPT.md's example repo layout (§36) and memory-bank file list (§17) are illustrative, not exhaustive | Architectural | Could cause drift if followed too literally once real structure is chosen | Treat as example; record actual structure decisions as ADRs, update [03-system-architecture.md](03-system-architecture.md) |
| `npm audit` (2026-07-17) reports transitive advisories in the NestJS 10.x/Express chain and Next.js 14.x that only clear via a major-version bump (Nest 11, Next 16) | Dependency | Known CVEs (DoS-class, one critical) remain until upgraded | Deliberately deferred rather than force-upgrading mid-feature (`npm audit fix --force` would pull in untested majors); revisit as a dedicated upgrade task, re-run full test suite before merging |
| Auth/Projects/Planning/Execution/Sync/AI/Analytics modules' integration/e2e tests (`services/auth/test/auth.e2e-spec.ts`, `services/projects/test/projects.e2e-spec.ts`, `services/planning/test/planning.e2e-spec.ts`, `services/execution/test/execution.e2e-spec.ts`, `services/sync/test/sync.e2e-spec.ts`, `packages/local-client/test/sync-roundtrip.e2e-spec.ts`, `services/ai/test/ai.e2e-spec.ts`, `services/analytics/test/analytics.e2e-spec.ts`) and the frontend Playwright specs (`apps/web/e2e/login.spec.ts`, `apps/web/e2e/projects.spec.ts`, `apps/web/e2e/planning.spec.ts`, `apps/web/e2e/execution.spec.ts`) require a real Postgres (Docker) and were not executed in the authoring sandbox (no Docker available there) | Testing | Full flows — including the Phase 3 closed-loop progress rollup, the Phase 4 start/complete/activity flow, the Phase 5 sync/conflict-resolution flow, and the Phase 7 owner-isolation guarantee — are untested against a live database/browser until first run | Written to spec and wired into `.github/workflows/ci.yml`, which does have Postgres; run locally via `docker compose -f infrastructure/docker/docker-compose.dev.yml up -d` then `npm run test:e2e -w @pee/auth` / `-w @pee/projects` / `-w @pee/planning` / `-w @pee/execution` / `-w @pee/sync` / `-w @pee/local-client` / `-w @pee/ai` / `-w @pee/analytics` / `-w web` before trusting them fully. **Exception (Phase 8):** `apps/desktop/e2e/desktop.spec.ts` needs no Postgres/Docker (pure SQLite/Electron) and was actually run in the authoring sandbox — it passed, including a real first-run SQLite bootstrap |
| No Prisma migration file has ever been generated (`packages/database/prisma/migrations/` doesn't exist) | Technical | `prisma migrate deploy` (used in CI and for prod) has nothing to apply — the database schema (including the Phase 5 composite indexes and Phase 7's `ExecutionEvent` index) has never actually been created against a real Postgres | `prisma migrate dev` needs a live database to diff against; run `npx prisma migrate dev --name init --schema packages/database/prisma/schema.prisma` once Docker/Postgres is available, commit the generated migration, then re-verify `migrate deploy` in CI |
| Project/goal/task **non-status-change** CRUD actions (create, rename, description edit) are not audit-logged | Security/Observability | No record of who created a project or renamed a goal/task, unlike auth events or status transitions | `ExecutionEvent` (Phase 4) now covers status transitions on `Goal`/`Task` unconditionally; a general domain-entity audit trail covering *all* CRUD (including non-status fields, and `Project` entirely) is deferred until a real need (compliance, support debugging) identifies what to capture — see [27-backlog.md](27-backlog.md) |
| No automated test exercises a real network call to the live Anthropic/OpenAI APIs (Phase 6) | Testing | `services/ai`'s actual wire compatibility with either vendor's current API is unverified — `ai.e2e-spec.ts` proves the HTTP/persistence/domain-integration plumbing via an in-test fake-provider override, and unit specs mock the vendor SDKs | Run a manual smoke test with real vendor credentials (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`) before relying on this feature in a live environment — normal posture for third-party LLM integrations, not a blocker for calling Phase 6 complete |
| `AIRecommendation` (Phase 6) is not covered by the Phase 5 sync protocol | Technical | A future offline UI can't show pending/past AI suggestions while disconnected | Same precedent as `TaskExecutionSession`/`ExecutionEvent` — no offline UI exists yet to need it; the sync registry's bidirectional-only shape would need rework to add a 4th entity. See [27-backlog.md](27-backlog.md) |
| Analytics (Phase 7) computes every metric via live query aggregation, no materialized rollup table | Technical | At real scale (thousands of `TaskExecutionSession`/`ExecutionEvent` rows per user), the bounded date-range queries stay cheap, but this hasn't been load-tested | Deliberately chosen over a summary table or scheduled job — current per-user data volumes are small and adding that infrastructure now would be speculative (Sustainable Complexity, Principle 8). Revisit if real usage data shows query cost becoming an issue — see [27-backlog.md](27-backlog.md) |

Product-code risks begin with Phase 1 (Authentication), implemented 2026-07-17; Phase 2 (Projects), Phase 3 (Planning Engine), Phase 4 (Execution Engine), Phase 5 (Memory Engine), Phase 6 (AI Integration), Phase 7 (Analytics), and Phase 8 (Desktop) added 2026-07-18. Add entries here as soon as new issues are discovered; never leave a known risk undocumented.
