## Latest handoff — 2026-07-18 (Phase 10)

**Current phase:** 10 — Enterprise. **Complete — the last phase currently defined.**

**Completed work:** Implemented multi-tenancy, RBAC, and SSO against the Phase 1-9 foundation (`adr/0009`). New `Organization`/`Membership`/`SsoIdentity` Prisma models; `Project`/`Goal`/`Task` gained `organizationId`; `User.passwordHash` became nullable (SSO-only users). Every user gets an invisible personal `Organization` + `OWNER` `Membership` at registration. New `services/organizations` (`@pee/organizations`) module: `OrganizationsService`, `MembershipManagementService` (invite/promote/remove, blocking removal of the last `OWNER`), `OrganizationRolesGuard`+`@RequireRole()`, REST endpoints under `/organizations`. `ProjectsService`/`GoalsService`/`TasksService` retrofitted: `findOwnedOrThrow` → `findAccessibleOrThrow` (membership-based — any `MEMBER`+ reads/updates, not-a-member still 404s), `create` resolves/inherits `organizationId`, `archive` gained `assertDestructivePermission` (creator or `ADMIN`/`OWNER`, else 403). Confirmed by tracing the actual code — not assumed — that `services/execution`/`services/ai`/`services/sync` needed **zero changes**, since all three delegate writes through those same three retrofitted services; `services/analytics` deliberately stays owner-scoped (documented boundary). SSO: OIDC via Auth.js's native `type: 'oidc'` provider (this backend only runs a secret-guarded provisioning endpoint after Auth.js's own verified exchange); SAML via a real SP (`@node-saml/node-saml`) behind a self-built OAuth2-authorization-code façade (`SamlBridgeService`/`SamlController`), since Auth.js has no native SAML provider type — both funnel through one shared `SsoProvisioningService`, both feature-flagged off by default. `apps/web` gained its first shared `dashboard/layout.tsx` shell (nav + a pure-UI org switcher, no server-side "active org" state), `/dashboard/organizations` + `/dashboard/organizations/[id]/members` pages, an organization selector on the new-project form, and conditional SSO login buttons.

**Two real bugs were found and fixed during this session, not just designed around:** (1) A genuine circular package dependency — `@pee/auth` importing `@pee/organizations` while `@pee/organizations`'s controllers import `CurrentUser`/`JwtAuthGuard` from `@pee/auth` — caused an actual Node.js CommonJS `require()` cycle, confirmed by a runtime `TypeError: (0, auth_1.CurrentUser) is not a function` when first tried. Fixed by having `AuthService` read `Membership`/`Organization` directly via Prisma instead of calling into `@pee/organizations` at all — the same read-only-direct-Prisma carve-out `@pee/analytics` already uses. (2) A security review pass (self-initiated, not requested) found two real issues before considering the phase done: `SsoProvisionGuard`'s secret comparison used a plain `!==` (a timing side-channel — fixed with `crypto.timingSafeEqual`), and the SAML bridge's redirect-allowlist check used `redirectUri.startsWith(allowedOrigin)` (a classic open-redirect bypass — `https://app.example.com.evil.com` also "starts with" `https://app.example.com`; fixed by comparing parsed `URL.origin` values, extracted into its own testable `saml/redirect-allowlist.ts` file with a dedicated regression test).
